Enterprise-managed authorization is turned on once for your tenant. For what it is and how it fits next to the AI access management gateway, see the [enterprise-managed authorization overview](/product/admin/enterprise-managed-authorization/overview).

## Before you begin

- Enterprise-managed authorization is in early access. [Contact the C1 Support team](mailto:support@c1.ai) to have it enabled for your tenant.
- You need an admin role to change tenant settings.

## Enable enterprise-managed authorization for your tenant

Enabling it exposes the enterprise-managed authorization surfaces to admins and sets the tenant-wide defaults that apply to every MCP server you register afterward. It does not grant any user access to any server on its own. You still register each server, enable its scopes, and bundle them into access profiles before access becomes requestable.

Go to **Settings** in the left navigation, open the **AI** section, and select **Cross-App Access**. The page is read-only until you click **Edit** (top right). Make your changes, then click **Save**. The edit view shows the **Enable cross-app access** toggle, the **Default signing algorithm** and **Maintained signing algorithms** controls, and the **Default grant lifetime** picker — work through them in the order below.

<Steps>
<Step>
Turn on the **Enable cross-app access** toggle.

This records your intent to run the issuer. As the page notes, the issuer is live only when this is on *and* C1 has enabled the operator rollout for your tenant.
</Step>
<Step>
Set the **Default signing algorithm**, and choose which algorithms C1 maintains keys for.

There are two related controls:

- **Default signing algorithm** — a single choice applied to any system that doesn't override it. The options are **ES256 (default)**, **EdDSA**, and **RS256**.
- **Maintained signing algorithms** — the set of algorithms C1 keeps signing keys for. **EdDSA** is always maintained (shown checked and locked); **RS256** and **ES256** are checkboxes you can turn on.

The MCP server you issue tokens for verifies C1's signature at its own authorization server, so choose a default that server can verify, and make sure that algorithm is maintained. ES256 is the default and the safest choice for broad compatibility, and RS256 is widely supported. EdDSA is supported, but verification for it is uneven across servers, so confirm the target server before you rely on it.
</Step>
<Step>
Set the **Default grant lifetime**.

This is a duration picker. Leave it on **Indefinite** to inherit C1's built-in default, or choose a preset (**1 hour**, 12 hours, 1 day, 1 week, 2 weeks, 1 month) or a **Custom** value. Custom durations are expressed in hours, days, weeks, or months. Here **Indefinite** means "inherit C1's built-in default," which is **300 seconds (5 minutes)** — it does *not* mean tokens never expire. This default applies to any resource server that doesn't set its own **Maximum grant lifetime**. A token keeps working until it expires, so a shorter lifetime tightens the window in which a revoked grant still has effect.
</Step>
</Steps>

## How enterprise-managed authorization is turned on

Enterprise-managed authorization is enabled at the tenant level on this page. Once it's on, the surfaces for registering MCP servers, enabling scopes, and reviewing the issuance audit appear for admins.

Enabling it at the tenant does not turn anything on for individual servers. You register each server one at a time and choose when it goes live. Later, you can pause an individual server to stop new token requests for it without removing its configuration. Pausing one server does not affect the others, and re-enabling it restores its existing setup.
