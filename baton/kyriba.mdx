## Gather Kyriba credentials

<Warning>
To configure the Kyriba connector, you need administrator access to the Kyriba Developer Portal to create API credentials.
</Warning>

The connector authenticates using OAuth2 client credentials. You will need:

- **Kyriba URL** — the base URL of your Kyriba instance, including the tenant-specific gateway path (for example, `https://api.<tenant>.kyriba.com/api`). This is **not** the bare host — it must include the gateway path prefix that your tenant uses (such as `/api` or `/gateway/api`).
- **Token URL** — the OAuth2 token endpoint (for example, `https://auth.<tenant>.kyriba.com/oauth/token`)
- **Client ID** — your OAuth2 client identifier
- **Client Secret** — your OAuth2 client secret

<Info>
The exact host and gateway path vary by Kyriba tenant. Check your Kyriba Developer Portal or Kyriba administrator for the correct API base URL and token endpoint for your environment.
</Info>

<Steps>
  <Step>
    Log in to the [Kyriba Portal](https://www.kyriba.com) and navigate to **API Access** or **OAuth Applications**.
  </Step>

  <Step>
    Create a new OAuth2 application or API credential set named `ConductorOne`.
  </Step>

  <Step>
    Grant the following OAuth scopes to enable full sync and provisioning:

    | Scope | Used for |
    | :--- | :--- |
    | `user-scope` | Sync users and access profiles; create, update, and delete users; assign data permission profiles |
    | `user-group-scope` | Sync user groups; add and remove members |
    | `data-permission-profile-scope` | Sync data permission profiles |

    Within each scope, make sure the OAuth2 client has both **read** and **write** permissions enabled. The table below shows which API permissions are required for each ConductorOne capability:

    | ConductorOne capability | Kyriba API endpoint | Required scope | Permission level |
    | :--- | :--- | :--- | :--- |
    | Sync users and access profiles | `GET /v1/users` | `user-scope` | Read |
    | Sync data permission profiles | `GET /v1/data-permission-profiles` | `data-permission-profile-scope` | Read |
    | Sync user groups | `GET /v1/user-groups` | `user-group-scope` | Read |
    | Create user | `POST /v1/users` | `user-scope` | Write |
    | Update user (data permission profile assignment) | `PUT /v1/users/{ref}` | `user-scope` | Write |
    | Delete user | `DELETE /v1/users/{ref}` | `user-scope` | Write |
    | Add/remove user group members | `POST/DELETE /v1/user-groups/{ref}/users` | `user-group-scope` | Write |

    <Warning>
    If the OAuth2 client has read-only permissions on `user-scope`, sync and account creation may succeed but user update (used for data permission profile assignment) and user delete will fail with `403 Forbidden`. Ensure **write** permissions are enabled for all scopes that correspond to provisioning capabilities you intend to use.
    </Warning>
  </Step>

  <Step>
    Copy the **Client ID**, **Client Secret**, and **Token URL**. Save these securely — the secret cannot be retrieved again after creation.
  </Step>
</Steps>

    Create two Kubernetes manifest files for your Kyriba connector deployment:

    #### Secrets configuration

    ```yaml expandable
    # baton-kyriba-secrets.yaml
    apiVersion: v1
    kind: Secret
    metadata:
      name: baton-kyriba-secrets
    type: Opaque
    stringData:
      # C1 credentials
      BATON_CLIENT_ID: <C1 client ID>
      BATON_CLIENT_SECRET: <C1 client secret>

      # Kyriba credentials
      BATON_KYRIBA_URL: https://api.<tenant>.kyriba.com/api
      BATON_TOKEN_URL: https://auth.<tenant>.kyriba.com/oauth/token
      BATON_KYRIBA_CLIENT_ID: <Kyriba OAuth2 client ID>
      BATON_KYRIBA_CLIENT_SECRET: <Kyriba OAuth2 client secret>
    ```

    See the connector's README or run `--help` to see all available configuration flags and environment variables.

    #### Deployment configuration

    ```yaml expandable
    # baton-kyriba.yaml
    apiVersion: apps/v1
    kind: Deployment
    metadata:
      name: baton-kyriba
      labels:
        app: baton-kyriba
    spec:
      selector:
        matchLabels:
          app: baton-kyriba

        metadata:
          labels:
            app: baton-kyriba

            image: public.ecr.aws/conductorone/baton-kyriba:latest

            - name: BATON_HOST_ID
              value: baton-kyriba
            envFrom:
            - secretRef:
                name: baton-kyriba-secrets
    ```

## Troubleshooting

**"expected a JSON response but received something else"** — This error means the connector received an HTML page instead of a JSON API response. It almost always indicates that the **Kyriba URL** is misconfigured. Verify that:
- The URL includes your tenant's gateway path prefix (for example, `/api` or `/gateway/api`) — a bare host without the path prefix will return an HTML page.
- The hostname resolves and is correct for your Kyriba environment.
- You can reach `<your-kyriba-url>/v1/users` from the connector's network and receive a JSON response.

**`403 Forbidden` on user update or delete** — Sync and user creation may succeed while update (`PUT /v1/users/{ref}`) and delete (`DELETE /v1/users/{ref}`) operations fail with `403`. This means the OAuth2 client has read-only permissions on the `user-scope`. In the Kyriba Developer Portal, ensure the client has **write** permissions on `user-scope` to enable all provisioning capabilities.

## Known Limitations

**Access profiles with no assigned users** — The Kyriba API does not expose a standalone list endpoint for access profiles. ConductorOne derives access profile resources from user-assignment data. Any access profile that exists in Kyriba but has no users currently assigned to it will not appear in the ConductorOne catalog until at least one user is assigned to it and a sync completes.
