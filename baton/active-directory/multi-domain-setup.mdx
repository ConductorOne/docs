This guide covers the prerequisites and configuration for syncing users, groups, and gMSAs across multiple Active Directory domains or forests using the `additional-domains` config setting.

## Overview

The connector can sync resources from a primary domain plus one or more additional trusted domains. Each domain is queried independently via LDAP. Cross-domain references (group members from remote domains, foreign security principals, gMSA SID grants) are automatically resolved.

**Supported trust types:**
- 1-way outbound trust (primary trusts remote, or remote trusts primary)
- 2-way trust
- Cross-forest trust
- External (non-transitive) trust

## Prerequisites

### 1. Active Directory trust

A trust relationship must exist between the domains. The connector machine (where the connector runs) must be in a domain that either:
- **Trusts** the remote domain (can authenticate users from the remote domain), OR
- **Is trusted by** the remote domain (its Kerberos tickets are accepted by the remote domain's DCs)

For Kerberos authentication to the remote domain, the connector's domain must be trusted by the remote domain. For simple bind (username/password), any network connectivity is sufficient.

To verify the trust:

```powershell
# From the primary domain's DC
Get-ADTrust -Filter * | Select-Object Name, Direction, TrustType, ForestTransitive
```

To create a forest trust (if one doesn't exist):

```powershell
# From the primary domain's DC â€” creates a 2-way forest trust
# You'll need Enterprise Admin credentials in both forests
netdom trust PrimaryDomain.com /d:RemoteDomain.com /add /twoway /realm
```

### 2. DNS resolution

The connector machine must be able to resolve the remote domain's DNS names. This is required for:
- Locating domain controllers (`_ldap._tcp.dc._msdcs.<domain>` SRV records)
- Connecting to specific DCs by hostname
- Kerberos ticket requests (KDC lookup via `_kerberos._tcp.<domain>`)

**Option A: Conditional forwarder (recommended)**

```powershell
# On the primary domain's DNS server
Add-DnsServerConditionalForwarderZone `
    -Name "Remote-Forest-01.lan" `
    -MasterServers "10.0.2.10" `
    -ReplicationScope "Forest"
```

**Option B: Stub zone**

```powershell
Add-DnsServerStubZone `
    -Name "Remote-Forest-01.lan" `
    -MasterServers "10.0.2.10" `
    -ReplicationScope "Forest"
```

**Option C: Secondary zone** (full zone transfer â€” for air-gapped or high-availability setups)

To verify DNS resolution:

```powershell
# Must return SRV records for the remote domain
Resolve-DnsName -Name "_ldap._tcp.dc._msdcs.Remote-Forest-01.lan" -Type SRV

# Must resolve the remote DC's hostname
Resolve-DnsName -Name "DC01.Remote-Forest-01.lan"

# Must resolve the remote domain's A record (or at least its DCs)
nslookup Remote-Forest-01.lan
```

### 3. Network connectivity

The connector machine must reach the remote domain's DCs on the required ports:

| Port | Protocol | Purpose |
|------|----------|---------|
| 389  | TCP/UDP  | LDAP (plaintext with signing) |
| 636  | TCP      | LDAPS (TLS-encrypted LDAP) |
| 88   | TCP/UDP  | Kerberos KDC (for external/Kerberos auth) |
| 3268 | TCP      | Global Catalog (optional, for GC queries) |
| 3269 | TCP      | Global Catalog over SSL (optional) |

To test connectivity:

```powershell
# Test LDAP
Test-NetConnection -ComputerName "DC01.Remote-Forest-01.lan" -Port 389

# Test LDAPS
Test-NetConnection -ComputerName "DC01.Remote-Forest-01.lan" -Port 636

# Test Kerberos
Test-NetConnection -ComputerName "DC01.Remote-Forest-01.lan" -Port 88
```

### 4. LDAPS certificate trust (if using LDAPS)

If connecting to the remote domain over LDAPS (port 636), the connector machine must trust the remote domain's CA certificate. Without this, TLS handshakes will fail.

**Option A: Cross-forest CA trust via AD (automatic)**

If both forests have Enterprise CAs and a 2-way trust, the remote forest's root CA certificate is typically published to AD automatically. Verify:

```powershell
# Check if the remote CA cert is in the NTAuth store
certutil -viewstore "ldap:///CN=NTAuthCertificates,CN=Public Key Services,CN=Services,CN=Configuration,DC=leet-lab,DC=lan,DC=minn,DC=info"
```

**Option B: Manual CA certificate import**

Export the remote forest's root CA certificate and import it on the connector machine:

```powershell
# On the remote domain's CA server â€” export the root CA cert
certutil -ca.cert remote-root-ca.cer

# On the connector machine â€” import into Trusted Root CAs (machine store)
Import-Certificate -FilePath "remote-root-ca.cer" `
    -CertStoreLocation "Cert:\LocalMachine\Root"
```

**Option C: Skip certificate verification in go-ldap mode (lab/testing only)**

In go-ldap mode (`mode: ldap`), the `ldaps-skip-verify` flag controls TLS certificate validation. It defaults to `true`, so self-signed certificates work without importing the CA. For production, set `ldaps-skip-verify: false` and import the CA cert as shown above.

<Note>
`ldaps-skip-verify` has no effect in WinLDAP mode. WinLDAP always validates certificates against the Windows certificate store. You must import the remote CA cert via Option B above.
</Note>

**Option D: Use plaintext LDAP with signing**

For lab environments where you can't set up proper CA trust and are using WinLDAP mode, you can connect over plain LDAP (port 389) with signing instead of LDAPS. Omit the `ldaps: true` field from the additional domain config.

To verify LDAPS connectivity to a remote DC:

```powershell
$tcp = New-Object System.Net.Sockets.TcpClient("DC01.Remote-Forest-01.lan", 636)
$ssl = New-Object System.Net.Security.SslStream($tcp.GetStream(), $false, {$true})
$ssl.AuthenticateAsClient("DC01.Remote-Forest-01.lan")
Write-Host "LDAPS working! Protocol: $($ssl.SslProtocol), Cert: $($ssl.RemoteCertificate.Subject)"
$ssl.Close(); $tcp.Close()
```

You can also use the connector's built-in diagnostic to test all domains at once. See [LDAPS setup](/baton/active-directory/ldaps-setup) for details:

```console
./baton-active-directory test-ldaps
```

### 5. Kerberos configuration (for external/Kerberos auth)

When using the default authentication mode (`bind-type: external` / Kerberos), the connector machine's Kerberos configuration must be able to obtain tickets for the remote domain.

**On Windows (domain-joined):** Kerberos cross-realm trust is typically automatic when a forest trust or external trust exists. The connector requests a referral ticket from the local KDC, which issues a cross-realm TGT.

**Important SPN note:** When connecting to the remote DC, the connector targets the DC's **hostname** (for example, `DC01.Remote-Forest-01.lan`), not the domain FQDN. This ensures the Kerberos SPN (`ldap/DC01.Remote-Forest-01.lan`) resolves correctly. Targeting the domain FQDN can fail because the SPN is registered under the DC's machine account, not the domain itself.

To verify Kerberos tickets:

```powershell
# After running the connector, check that cross-realm tickets were obtained
klist

# You should see tickets for:
#   krbtgt/REMOTE-FOREST-01.LAN@LEET-LAB.LAN.MINN.INFO  (cross-realm TGT)
#   ldap/DC01.Remote-Forest-01.lan@REMOTE-FOREST-01.LAN   (service ticket)
```

**On Linux:** Cross-realm Kerberos requires `/etc/krb5.conf` entries for the remote realm:

```ini
[realms]
REMOTE-FOREST-01.LAN = {
    kdc = DC01.Remote-Forest-01.lan
    admin_server = DC01.Remote-Forest-01.lan
}

[domain_realm]
.remote-forest-01.lan = REMOTE-FOREST-01.LAN
remote-forest-01.lan = REMOTE-FOREST-01.LAN

[capaths]
LEET-LAB.LAN.MINN.INFO = {
    REMOTE-FOREST-01.LAN = .
}
```

### 6. Permissions

The account running the connector needs read access to the remote domain's directory. This is typically satisfied by:

- **Default**: The `Authenticated Users` group has read access to most AD objects. A domain trust ensures that users from the trusting domain are treated as authenticated users in the trusted domain.
- **Restricted environments**: If the remote domain has restricted read access, grant the connector's service account explicit read permissions on the OUs being synced.
- **gMSA sync**: Reading `msDS-GroupMSAMembership` attributes requires read access to the gMSA objects. Modifying them (Grant/Revoke) requires write access â€” set `read-only: false` in the additional domain config.

## Configuration

`additional-domains` is a **YAML-only** setting â€” it cannot be set via CLI flags or environment variables. It uses native YAML syntax (a list of maps), not JSON-encoded strings.

### Config file (YAML)

```yaml
# Primary domain
domain: leet-lab.lan.minn.info
base-dn: DC=leet-lab,DC=lan,DC=minn,DC=info
bind-type: external
ldaps: true
ldaps-port: 636
mode: winldap
sync-scope: Standalone
enable-gmsa-sync: true

# Additional domains
additional-domains:
  # Kerberos auth (default) â€” simplest config for trusted domains
  - domain: Remote-Forest-01.lan

  # With LDAPS
  - domain: Remote-Forest-01.lan
    ldaps: true
    ldaps-port: 636

  # Simple bind with explicit credentials
  - domain: other.example.com
    bind-type: simple
    bind-user: "CN=svc-baton,OU=Service Accounts,DC=other,DC=example,DC=com"
    bind-password: "SecretPassword123"

  # Read-only with explicit base DN (skip auto-discovery)
  - domain: partner.corp
    base-dn: "DC=partner,DC=corp"
    read-only: true
```

### Additional domain config fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `domain` | string | (required) | FQDN of the remote domain |
| `base-dn` | string | (auto-discovered) | Base DN for LDAP searches. Auto-discovered from rootDSE `defaultNamingContext` if not specified. |
| `bind-type` | string | `"external"` | `"external"` for Kerberos/GSSAPI, `"simple"` for username/password |
| `bind-user` | string | | DN of the bind user (required for simple bind) |
| `bind-password` | string | | Password for simple bind |
| `ldaps` | `*bool` | (inherits primary) | Enable LDAPS (TLS) on port 636. Pointer type â€” omit to inherit the primary domain's setting; set `true` or `false` to override. |
| `ldaps-port` | int | `636` | LDAPS port (if non-standard) |
| `read-only` | bool | (see below) | Prevent write operations (Grant/Revoke) to this domain |
| `user-search-dn` | string | (base-dn) | Override the user search base for this domain |
| `user-search-filter` | string | (primary filter) | Override the user search filter for this domain |
| `group-search-dn` | string | (base-dn) | Override the group search base for this domain |
| `group-search-filter` | string | (primary filter) | Override the group search filter for this domain |
| `skip-ous` | list | | OUs to exclude in this domain |
| `only-ous` | list | | OUs to include in this domain (exclusive with `skip-ous`) |
| `gmsa-search-dn` | string | (base-dn) | Override the gMSA search base for this domain |
| `gmsa-search-filter` | string | (primary filter) | Override the gMSA search filter for this domain |

**`read-only` default behavior:** When omitted, defaults to `true` (read-only) unless `bind-type` is explicitly set, in which case it defaults to `false` (read-write). The logic is that providing domain-specific bind credentials signals the operator intends to allow writes. Set `read-only: true` or `read-only: false` explicitly to override this default.

### Auto-discovery

When `base-dn` is not specified, the connector queries the remote domain's rootDSE for `defaultNamingContext` and uses that as the search base. This works for most standard AD deployments.

## How it works

### Resource enumeration

During `List()`, the connector uses `SearchAllDomains()` to paginate across all configured domains. It completes the primary domain fully, then moves to each additional domain. Resources from all domains appear in a single unified sync.

### Cross-domain grant resolution

When resolving grants (group memberships, gMSA permissions), the connector encounters DNs and SIDs from remote domains:

- **DN-based references** (for example, group member DNs): `getObjectFromDN()` detects the `DomainRouter` and uses `SearchByDN()` to route the lookup to the correct domain based on the DN's suffix.
- **SID-based references** (for example, gMSA `msDS-GroupMSAMembership` DACL): `ResolveSIDCrossDomain()` tries each domain in order until a match is found.
- **Foreign Security Principals**: Objects in `CN=ForeignSecurityPrincipals` are resolved by their `objectSid` attribute against the remote domain.

### NetBIOS name resolution

NetBIOS domain names (for example, `LEET-LAB`, `REMOTE-FOREST`) are needed for sAMAccountName display. The connector queries each forest's `configurationNamingContext` independently for `crossRef` entries, ensuring names resolve for all forests.

On Windows, the primary domain uses the native `DsCrackNamesW` API; additional domains fall back to LDAP crossRef queries.

## Troubleshooting

### "LDAP Result Code 32 (No Such Object)" during grants sync

**Cause**: The connector is trying to resolve a DN or SID from a remote domain but can't reach it.

**Fix**: Ensure the additional domain is configured in `additional-domains` and that DNS/network connectivity is working.

### "baseDN auto-discovery failed" for additional domain

**Cause**: The connector can't reach the remote domain's rootDSE.

**Fix**: Check DNS resolution and network connectivity. Try specifying `base-dn` explicitly in the additional domain config.

### "DsCrackNamesW failed, falling back to LDAP crossRef"

**Cause**: The native Windows API couldn't resolve NetBIOS names. This is normal for remote forest domains â€” they fall back to LDAP crossRef queries automatically.

**Action**: No action needed. This is informational.

### Kerberos authentication fails to remote domain

**Symptoms**: "SASL bind failed" or "Cannot obtain Kerberos ticket" errors.

**Fix**:

1. Verify the trust exists: `Get-ADTrust -Filter *`
2. Verify DNS: `Resolve-DnsName _ldap._tcp.dc._msdcs.<remote-domain> -Type SRV`
3. Target the DC hostname, not the domain FQDN, in the domain config
4. Fall back to simple bind by setting `bind-type: simple`, `bind-user`, and `bind-password` in the domain's config entry

### "netbios domain is empty" warnings for remote domain objects

**Cause**: The NetBIOS cache failed to load crossRef entries from the remote domain.

**Fix**: Ensure the remote domain's LDAP is reachable and the connector has read access to the `configurationNamingContext`.

### LDAP error 81 ("Server Down") connecting to remote domain

**Cause**: This usually indicates a TLS handshake failure, most commonly because the remote forest's CA certificate is not trusted on the connector machine.

**Fix (WinLDAP mode)**: Import the remote forest's root CA certificate into the Windows Trusted Root CAs store:

```powershell
certutil -addstore Root <path-to-remote-ca.cer>
```

**Fix (go-ldap mode)**: Either import the CA certificate into the system trust store, or set `ldaps-skip-verify: true` in the config (not recommended for production).

**Diagnostics**: Run `./baton-active-directory test-ldaps` to see exactly which step fails (DC resolution, TCP, TLS, or bind).

### Objects from remote domain not appearing in sync

**Cause**: The additional domain wasn't initialized successfully during startup.

**Fix**: Check the startup logs for warnings about the additional domain. Run `./baton-active-directory test-ldaps` to diagnose connectivity. Common issues:
- DNS can't resolve the domain
- LDAPS handshake fails (certificate trust issue â€” see "LDAP error 81" above)
- Kerberos ticket request fails
- Simple bind credentials are wrong

### Performance considerations

Each additional domain adds LDAP queries to the sync cycle. For large domains:
- The connector completes one domain fully before moving to the next
- LDAP pagination (default 1000 entries per page) is used for each domain
- Total sync time scales linearly with the number of domains and objects
